## 2. V2 目标与分支目的

### 2.1 为何新建架构分支？

当前 `main` 实现积累了若干设计妥协：

- **认知与 `nanobot` 耦合**：`runtime_bridge.py` 直接包裹 `nanobot.agent.loop.AgentLoop`，导致推理层难以独立替换或测试。
- **业务逻辑与 LLM 交互逻辑混杂**：编排决策（选哪个角色、用哪个 backend）和提示词组装（如何与模型交互）没有明确分离。
- **Agent 框架被当作基础设施**：当前设计继承了 `nanobot` 的配置路径、CLI 组装方式和会话约定，使 `secnano` 依赖于它本不应拥有的上游设计决策。
- **缺少正式的入口解析层**：原始 CLI 字符串在到达编排器之前没有经过专门的解析/校验阶段。
- **缺少正式的输出守卫**：模型输出直接流向 I/O 层，没有显式的校验或安全检查边界。

### 2.2 V2 设计目标

1. **高内聚、低耦合**：每个模块拥有单一、明确的职责。模块之间通过显式的数据契约交互，而不是共享内部状态或直接导入彼此的内部实现。
2. **依赖倒置**：高层编排策略不依赖低层执行或 Provider 实现，两者都依赖共享的抽象契约。
3. **业务逻辑与 LLM 交互逻辑分离**：编排器决定"做什么"，认知内核决定"如何推理"，两者不相互侵入。
4. **将 Agent 框架视为库，而非框架**：`nanobot`（及未来任何 Agent SDK）以库的方式从认知模块内部调用，不拥有配置路径、CLI 组装或会话管理。
5. **拥抱标准协议与 Provider 抽象**：LLM Provider、执行 backend 和能力适配器均通过稳定的内部接口访问，替换任意一个不会引发跨系统级联变更。
6. **先设计后实现**：在本分支中，架构首先以文档形式固定，代码跟随文档，而非反之。

## 3. 设计原则详述

### 3.1 单一主控

系统向外界暴露唯一一个**主控 agent**（中书省）。  
主控本身由 **LLM 驱动**——它进行推理、维护记忆，并借助技能做出调度决策，而不是一个确定性规则引擎。  
六部中的角色执行容器由主控调度——它们不是并行的 agent。  
所有新能力都必须通过能力适配器层注册，并受单一主控统领。

当某个会话已有活跃的执行容器时，主控优先将请求路由到该容器，以保持执行上下文连续性。只有在容器失效、超时或策略明确要求时，才重新分配新容器。

### 3.2 稳定的数据契约

每个模块边界都以一组数据结构表达。  
模块产出输出时，产出的是约定的契约类型之一。  
模块消费输入时，消费的是约定的契约类型之一。  
任何模块都不直接访问另一个模块的内部状态。

### 3.3 短暂执行，持久资产

三类对象跨越不同的时间维度：

- **Role（角色）** 是永久的职责定义，在所有会话和容器实例中长期存在。
- **Container（容器）** 是短暂的运行时执行实例，受资源约束且可回收，不会超越其工作而存在。
- **Session（会话）** 是中期任务连续性对象，比单个容器更稳定，但比角色定义更具体。会话在活跃期间绑定到某个容器；绑定可在必要时迁移。

角色资产（SOUL、ROLE、MEMORY、技能、POLICY）和任务归档是系统长期连续性的载体。容器只保存临时运行态状态；所有持久状态必须在容器回收前写回归档与状态模块。

### 3.4 架构拓扑：朝廷与六部

V2 架构采用扩展的朝廷与六部隐喻。  
每个 V2 组件对应一个历史上的中国行政机构。  
该命名方案在所有 V2 设计文档中统一使用。

### 4.2 奏折 / 数据模型

**历史参考**：在中国帝制时代，奏折（zòuzhé）是官员与皇帝之间的标准通信格式，所有请求以已知的、结构化的格式送达和发出。

**在 V2 中的角色**：

这是**共享数据契约层**——系统的内部语言。它不是运行时模块，而是所有其他模块所依赖的不可变数据 schema 集合。

核心 schema：

| Schema | 说明 |
|--------|------|
| `InboundEvent` | 来自通政司的规范化原始输入 |
| `Task` | 编排器调度的一个工作单元 |
| `ExecutionRequest` | 发送给六部的具体请求 |
| `ExecutionResult` | 执行返回的结果 |
| `Reply` | 发送给用户的最终结构化回复 |
| `TaskArchiveRecord` | 已完成任务的持久化记录 |
| `SessionState` | 可恢复的编排器会话数据，包含当前绑定的 `ContainerRecord` |
| `ContainerRecord` | 运行中容器的标识、角色绑定、生命周期状态和 IPC 地址 |
| `RoleSpec` | 角色资产：SOUL、ROLE、MEMORY、技能、POLICY |
| `CapabilityDescriptor` | 已注册能力的标识与接口契约 |

设计规则：

- 所有 schema 均为**不可变值对象**（无方法，无逻辑）。
- 每次跨越模块边界时使用上述类型之一。
- 任何模块不得发明临时的 dict 或字符串作为跨模块数据。

**与原有模块的关系**：  
替代并扩展原七模块设计中的*任务/消息模型模块*（Module 2）。

### 4.3 通政司 (Tongzhengsi) — 入口 / 输入解析器

**历史参考**：通政司负责在奏折、请愿和报告到达皇帝之前接收并路由，确保任何格式不合规的内容不进入决策链路。

**在 V2 中的角色**：

本模块位于系统边界，是**唯一的外部输入入口**。它：

- 从 CLI、API、通道或其他任何外部来源接收原始输入。
- 将输入校验并规范化为强类型的 `InboundEvent`。
- 在格式不合规的输入到达编排器之前予以拒绝或隔离。
- 不执行任何业务逻辑、不调用模型、不进行角色选择。

输入：

- 原始 CLI 参数 / HTTP 请求体 / 通道消息载荷

输出：

- `InboundEvent`（成功时）
- 结构化错误响应（校验失败时）

约束：

- 不得调用 LLM。
- 不得做出编排决策。
- 不得访问角色资产或归档。

**与原有模块的关系**：  
替代原设计中*输入输出模块*（Module 1）的入口部分。V2 将入口解析提升为正式的一级模块，并设置明确的校验门。

### 4.4 中书省 (Zhongshu Sheng) — 主控 + 大脑

**历史参考**：中书省是帝国朝廷最高行政机构，负责起草诏令、协调六部，将皇帝意志转化为具体命令。

**在 V2 中的角色**：

中书省是系统中**唯一对外暴露的主控 agent**，同时兼任全局调度枢纽和核心推理大脑。它由 LLM 驱动——主控本身由语言模型提供动力，而不是一个确定性规则引擎。

#### 4.4.1 职责

- 接收来自通政司的所有 `InboundEvent` 并完成任务理解。
- 维护主控的全局上下文、会话状态和系统级规则。
- 根据任务类型、角色能力、历史状态和当前上下文做出调度决策。
- 决定任务应：
  - 直接回复。
  - 继续追问以澄清。
  - 委派给六部中的具体执行容器。
- 跟踪任务状态，接收 `ExecutionResult`，并决定是否继续委派、终止或汇总最终回复。

#### 4.4.2 能力

中书省不是无状态的调度器，它拥有：

- **自有记忆模块**：保存系统级长期规则、跨任务经验、调度历史和必要摘要。这是主控的记忆——与翰林院/史馆管理的角色记忆相互独立。
- **自有技能集合**：用于完成任务拆解、角色选择、状态检查、结果验收和续派决策的技能。
- **主控级工具**：主控可用于编排目的的工具（例如：检查会话状态、查询归档、检查容器可用性）。

#### 4.4.3 会话路由原则

中书省对携带 `Session` 的请求采用以下路由原则：

- 对于已有活跃 `Session` 的请求，主控**应优先将任务路由到该会话当前绑定的容器**。
- 这保持了执行上下文的连续性，避免了频繁切换容器所导致的状态丢失、记忆割裂和执行漂移。
- 只有以下情况发生时，主控才**重新为会话分配新容器**：
  - 原容器已失效、被回收或超时。
  - 原容器资源受限。
  - 策略明确要求迁移到不同容器。

需要重新分配时，主控在将会话绑定到新容器之前，先从归档与状态模块恢复必要上下文。

#### 4.4.4 边界

- 中书省负责**全局调度与委派决策**。
- 它**不**直接承担六部内部的具体执行步骤。
- 它**不**介入容器内部的步骤级调度、工具顺序选择或中间执行细节。
- 一旦任务进入执行容器，具体执行过程由容器内部运行的角色官员完成。

**与原有模块的关系**：  
合并并优化了原七模块设计中的*编排调度模块*（Module 3）和*认知内核模块*（Module 4）。V2 明确了主控由 LLM 驱动的本质，新增了会话路由原则，并在主控决策和容器执行之间划定了清晰边界。

### 4.5 翰林院 / 史馆 (Hanlin Academy / Shiguan) — RAG + 长期记忆

**历史参考**：翰林院是帝国的学者院，负责起草文件、保存机构知识，并就先例提供建议。史馆是保存官方历史记录的机构。

**在 V2 中的角色**：

本模块提供系统的**检索与长期记忆层**：

- 存储和检索角色记忆及提炼洞见。
- 在已归档任务记录、角色知识库和文档上提供 RAG（检索增强生成）索引。
- 向中书省认知子层暴露查询接口，以相关历史上下文丰富提示词。

输入：

- 来自认知子层的查询："关于 X 我知道什么？"
- 来自编排器的记忆提升请求："将此洞见提升至角色 Y 的长期记忆。"

输出：

- 检索到的上下文片段（用于 RAG 注入提示词）。
- 记忆提升确认。

设计规则：

- 记忆层不做编排决策。
- 记忆提升在写入长期记忆之前必须通过显式的基于规则的过滤器。
- 原始执行结果**不得**直接写入长期记忆。

**与原有模块的关系**：  
从原七模块设计中*角色与能力资产模块*（Module 5）和*归档与状态模块*（Module 7）中提取并正式化了分散的记忆与检索关注点。

### 4.6 六部 (Liubu) — 角色执行体系

**历史参考**：六部是帝国政府的行政分支：吏、户、礼、兵、刑、工。各部均有具体官员承担具体的政府工作。

**在 V2 中的角色**：

六部是系统的**角色执行体系**。六部中的执行官员不是静态角色定义本身——他们是挂载角色资产并运行于隔离容器中的**角色执行实例**。

#### 4.6.1 职责

- 承接中书省委派的任务。
- 在隔离执行环境中完成任务细化、步骤规划、工具调用与结果产出。
- 维持局部执行上下文，持续处理同一 `Session` 下的后续命令。
- 在任务完成后回写执行结果、工件索引、状态摘要与记忆候选。

#### 4.6.2 执行实体定义

六部中的"官员"是**运行时容器实例**，而不是抽象角色文件。

- 每个执行容器启动时**必须挂载**所属角色的相关资产。
- 角色资产至少包括：职责说明、提示词、长期记忆、技能、策略、工具权限和必要工作目录。
- 容器不是单纯的命令执行沙箱——它是**角色执行官员的运行载体**，拥有自己的 LLM、工具、记忆和局部状态。

#### 4.6.3 容器内执行原则

一旦任务进入容器，**具体执行过程由容器内运行的 LLM 完成**。

容器内 LLM 负责：

- 任务细化
- 步骤规划
- 工具调用
- 中间结果判断
- 结果生成与工件产出

外部主控（中书省）只负责："**是否委派、委派给谁、资源边界是什么**"。

容器内部负责："**具体如何完成任务**"。

#### 4.6.4 并行控制

六部执行体系的并行能力通过**活跃容器的最大数量**控制：

- 每个活跃容器代表一个可用执行官员席位。
- 系统可按全局或按角色设置容器上限，以限制同一时刻的并发执行规模。
- 超出上限的任务应进入等待、排队或延迟调度状态，而不是无限制扩容。

#### 4.6.5 通信与生命周期

- 执行容器通过 **IPC** 与外部系统交互。
- IPC 用于：接收新任务、响应状态查询、接收补充上下文、处理中断信号和接收终止命令。
- 容器应被视为**可持续交互的执行单元**，而不是一次性脚本进程。
- 同一容器在其生命周期内可以持续处理同一 `Session` 的后续任务。
- 容器在任务完成、主动退出或长期不活跃时应被回收。
- **回收前，容器必须回写**：必要状态摘要、关键工件索引、执行结果和可提升记忆至归档与状态模块。

子组件：

| 子组件 | 职责 |
|--------|------|
| 容器 Backend | 角色容器生命周期：启动、挂载资产、IPC、回收 |
| Host Backend | 进程内直接执行（用于轻量级任务） |
| 工具注册表 | 工具定义、执行调度、权限执行 |
| 工件管理器 | 工作区挂载、工件收集、结果打包 |

#### 4.6.6 边界

- 六部执行体系负责执行落地，不负责全局调度。
- 执行容器**不**直接对外暴露为新的用户入口。
- 所有六部官员都必须服务于中书省统一调度，而不是形成并行的第二套主控系统。

**与原有模块的关系**：  
对应原设计中的*执行模块*（Module 6）和*工具注册模块*。V2 明确了容器是由 LLM 驱动的角色执行实例（而非简单命令执行器），新增了并行控制机制，并正式化了基于 IPC 的生命周期。

### 4.7 门下省 (Menxia Sheng) — 校验器 / 输出守卫

**历史参考**：门下省是负责审查和驳回（拒绝副署）中书省起草的诏令在生效之前的审议机构，是对政府最高层输出的显式检查。

**在 V2 中的角色**：

本模块是**输出校验与安全门**：

- 接收来自中书省的草稿 `Reply` 对象。
- 应用策略规则、安全过滤器和格式校验。
- 通过审批后将回复传递给 I/O 层，或拒绝并返回结构化的退回意见。

输入：

- 来自中书省的草稿 `Reply`。
- 来自角色资产和平台规则的策略配置。

输出：

- 已审批的 `Reply`（发送至 I/O 输出层）。
- 结构化的拒绝（退回中书省修订或上报）。

设计规则：

- 门下省不生成内容，只负责审核。
- 策略规则为声明式，不内嵌 LLM 调用。
- 输出守卫在认知完成**之后**、回复发送给用户**之前**运行。

**与原有模块的关系**：  
这是 V2 中的**新正式模块**。在原七模块设计中，输出校验是隐式的（属于 I/O 模块或编排器的一部分）。V2 将其变为显式的、可测试的边界。

### 4.8 Role / Container / Session — 三层关系

`Role`、`Container`、`Session` 代表三种不同层级的对象，三者必须严格区分。

#### 4.8.1 Role（角色）

`Role` 是**静态角色定义**，表示一个执行官员的职责边界与能力画像。

包含：

- 职责说明
- 提示词
- 长期记忆
- 技能
- 工具权限
- 资源范围
- 安全策略

约束：

- `Role` 本身不是运行实例。
- 同一个 `Role` 可以被多个执行容器复用。
- `Role` 是长期稳定的职责身份，不直接等于某次具体执行过程。

#### 4.8.2 Container（容器）

`Container` 是某个 `Role` 在运行时的**实体化载体**，是实际承接任务并维持局部执行上下文的动态执行实体。

特征：

- `Container` 启动时挂载所属 `Role` 的相关资产。
- `Container` 在生命周期内保存运行态上下文、临时状态和执行工件。
- `Container` 是资源受限且可回收的，不等同于长期身份。
- `Container` 是实际完成任务的执行官员实例。

约束：

- 一个 `Container` 只从属于一个 `Role`。
- 一个 `Role` 可以对应多个 `Container` 实例。
- `Container` 是短期存在的执行实例，不应被当作永久身份对象。

#### 4.8.3 Session（会话）

`Session` 是**任务连续性的逻辑标识**，用于保证多轮请求在逻辑上属于同一工作链路。

特征：

- `Session` 表示一段连续任务处理过程的上下文连续体。
- `Session` 不直接等于 `Role`。
- `Session` 也不直接等于 `Container`。
- `Session` 由中书省维护，用于把多轮请求路由到合适的执行上下文。

默认绑定关系：

- 一个活跃 `Session` 在任一时刻应优先绑定到一个具体 `Container`。
- 同一个 `Session` 的后续请求，原则上应继续发送到其当前绑定的 `Container`。
- 该绑定用于保持任务现场、局部记忆和执行中间状态连续。

迁移规则：

- 当原 `Container` 失效、被回收、超时或不再适合当前任务时，中书省可以为该 `Session` 重新分配新的 `Container`。
- 重新分配时，应通过归档与状态模块恢复必要上下文。
- 重新分配后，可以继续沿用原有 `Role`，也可以依据任务变化切换到新的 `Role`。

长期与短期边界：

| 对象 | 性质 | 稳定性 |
|------|------|--------|
| `Role` | 长期稳定的职责身份 | 永久——跨所有会话和容器实例存在 |
| `Container` | 短期存在的运行时执行实例 | 短暂——空闲或完成时回收 |
| `Session` | 介于两者之间的任务连续体 | 中期——比单个容器更稳定，比角色定义更具体 |

## 7. 与 `nanobot`、`pyclaw`、`nanoclaw` 的关系

### 7.1 `nanobot`

在 V2 架构下，`nanobot` 被视为**可选库**，而非宿主框架。

- 中书省认知子层**可以**以库的方式调用 `nanobot.agent.loop.AgentLoop`，包裹在稳定的内部接口之后。
- 其他任何模块不得导入或依赖 `nanobot` 内部实现。
- `nanobot.config.*`、`nanobot.cli.*`、`nanobot.channels.*` 和 `~/.nanobot` 约定在 V2 中明确**排除在外**。
- 仓库中的 `packages/nanobot/` 是兼容垫片的存放位置，其范围严格限于认知子层垫片。

### 7.2 `pyclaw`（`refs/pyclaw`）

`pyclaw` 是容器运行时和 IPC 协议的**参考实现**。

- 六部容器 backend 应遵循 `refs/pyclaw` 中定义的协议。
- 不整体复制生产代码；而是研究协议并在 V2 中原生实现。

### 7.3 `nanoclaw`（`refs/nanoclaw`）

`nanoclaw` 是 TypeScript 容器编排的**参考实现**。

- 为容器生命周期、组队列、任务调度器和 IPC 提供架构参考。
- `refs/nanoclaw/docs/` 包含可参考的额外架构文档。
- 与 `pyclaw` 一样，仅作为参考材料，不导入代码。

### 7.4 汇总

| 组件 | V2 处理方式 |
|------|------------|
| `nanobot` 运行时循环 | 仅在中书省认知子层内作为库垫片调用 |
| `nanobot` 配置 / CLI / 通道 | **从 V2 中排除** |
| `refs/pyclaw` | 六部容器 backend 的协议参考 |
| `refs/nanoclaw` | 六部容器编排的架构参考 |
| `packages/nanobot/` | 仅认知兼容垫片的存放位置 |

## 8. `architecture-v2` 做什么与不做什么

### 本分支做

- 以文档形式固定参考架构。
- 定义模块边界、数据契约和依赖规则。
- 将当前代码映射到新架构（见 `../../mapping/main-to-v2-mapping.zh-CN.md`）。
- 规划分阶段实现路线（见 `../../plan/roadmap.zh-CN.md`）。

### 本分支不做

- 批量移植现有 `secnano/` 实现。
- 重写 `runtime_bridge.py` 或任何其他生产文件。
- 整体迁移 `nanobot` 集成。
- 在认知接口稳定之前锁定单一 Agent 框架。
- 修改 `main` 分支。
